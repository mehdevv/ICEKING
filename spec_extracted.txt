PART ONE — TECHNICAL SPECIFICATION
01  Project Overview
What we are building and why

1.1  Purpose
LoyalQR is a web-based digital loyalty platform for retail shops. It replaces paper stamp cards with QR-powered digital fidelity cards, tracks every purchase through employee scanning, detects fraud automatically, and provides the owner with full analytics and customer CRM — all from a single platform, with no mobile app required for customers.

05  Fraud Detection Engine
Rules, logic flow, and audit trail

5.1  Fraud Rules — Applied on Every Scan
Rule ID
Rule Name
Logic
Action on Trigger
FR-01
Max Daily Scans
Same customer scanned > N times today (N configurable, default 2)
BLOCK — stamp not awarded
FR-02
Employee-Customer Cooldown
Same employee + same customer within T minutes (default 5 min)
BLOCK — stamp not awarded
FR-03
Inactive Card
Customer card flagged isActive=false by admin
BLOCK — always
FR-04
Invalid QR Token
QR JWT signature invalid or expired
BLOCK — always
FR-05
Rapid Sequential Scans
Same customer scanned by any employee within 60 seconds
BLOCK — possible device sharing
FR-06
Employee Self-Scan
Employee scans their own fidelity card (if they have one)
BLOCK — always

5.2  Scan Processing Flow
All checks run server-side in a single atomic transaction. The client receives only APPROVED or BLOCKED — never partial state.

Request arrives at POST /api/scans with employee session + QR token + productIds
Verify JWT session — reject if invalid or insufficient role
Decode and verify QR token signature — FR-04
Load FidelityCard; check isActive — FR-03
Run FR-01: count today's approved scans for this customer
Run FR-02: check employee+customer within cooldown window
Run FR-05: check any scan in last 60s for this customer
Run FR-06: check if scanned card belongs to employee
If any rule triggered: insert FraudEvent, return BLOCKED response
If all clear: increment stampCount on FidelityCard
Check if stampCount >= shop threshold — if so: create Reward(PENDING), notify via response flag
Insert Scan record (APPROVED), insert ScanItems for each product
Return JSON: { status: APPROVED, newStampCount, rewardTriggered }

15  Screen-Specific Design Notes
Key UX decisions per screen

15.1  Employee Scanner Screen
The scanner screen is the most performance-critical UI in the app. It must work reliably on low-end Android devices with poor cameras in dimly lit shops.

Viewfinder: full-width rectangle, 1:1 ratio, dark overlay with cutout. Scanning line animation (top to bottom, 2s loop)
On successful decode: haptic feedback (navigator.vibrate), immediate API call
Result state: full-screen banner — GREEN (#0E9F6E) for APPROVED, RED (#E02424) for BLOCKED
If APPROVED: show customer name, stamp count, reward triggered indicator
If BLOCKED: show fraud rule that triggered (plain language)
Product selector: bottom sheet drawer, scrollable list with checkbox, confirm button
Auto-reset to camera after 3 seconds post-result
One-thumb operable — all tap targets min 48x48px

15.2  Customer Fidelity Card
Stamp grid: visualised as filled / empty circles (e.g. 9 slots for threshold 9)
Stamp icon: custom per shop (default: star) — configurable in settings
QR code: displayed at 200x200px minimum, centred, with 16px quiet zone
Shop branding: logo in top-left, brand colour as background gradient
Download button: generates PNG of full card for saving to camera roll
Reward badge: animated gold badge overlaid when reward is pending

15.3  Admin Dashboard Overview
4 KPI cards in a 2x2 grid (mobile) or 4-column row (desktop): Total Clients, Scans Today, Rewards Pending, Fraud Alerts
Scan trend chart: area chart, 30 days, primary blue fill
Fraud alert banner: only shown if fraudCount > 0 today — orange/red, links to Fraud Events
Quick action buttons below KPIs: View Scan Log, Export Contacts, View Rewards

15.4  Fraud Events Screen
Default filter: unreviewed events only
Each row: timestamp, employee name, customer name, rule triggered (plain language badge), Review button
Review modal: shows full scan context + employee + customer history + action buttons (Mark Reviewed / Override Block)
Fraud rows highlighted with a left-border red stripe (border-l-4 danger)
